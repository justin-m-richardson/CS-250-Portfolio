# CS 250 Portfolio

This repository contains my portfolio submission for CS 250. The artifact included in this repository is my Sprint Review and Retrospective document from the final project. This project helped me understand how Agile and Scrum practices are used in software development, especially when working with a team, responding to user needs, and improving a product over time.

## Portfolio Artifact

The main artifact in this repository is my Sprint Review and Retrospective document. This assignment pulled together the Agile and Scrum practices I worked with throughout the course. It gave me a chance to reflect on the development process, team roles, communication, and how a Scrum team can improve after each sprint.

## Reflection

### How do I interpret user needs and implement them into a program? How does creating user stories help with this?

I interpret user needs by trying to understand what the user actually wants to accomplish. It is not just about adding random features or guessing what sounds good. A program should solve a real problem or make something easier for the user. Creating user stories helps with this because they explain the need from the user’s point of view. A user story shows who the user is, what they need, and why they need it. This helps keep the project focused on features that actually matter.

### How do I approach developing programs? What Agile processes do I hope to incorporate into my future development work?

I approach developing programs by breaking the work into smaller parts and focusing on one piece at a time. This makes the project feel more manageable and helps me catch problems earlier. In the future, I want to use Agile processes like sprint planning, user stories, regular check-ins, sprint reviews, and retrospectives. I like that Agile allows a team to adjust when something changes instead of being locked into one plan from the beginning. I think this will help me become a better developer because it keeps the focus on progress, feedback, and improving the project as I go.

### What does it mean to be a good team member in software development?

To me, being a good team member in software development means being dependable, communicating clearly, and doing your part to help the team succeed. A good team member should ask questions when something is unclear, give updates on their work, and be open to feedback. It also means understanding that your work affects the rest of the team. Software development is not just about writing code by yourself. It is about working with other people to build something that meets the user’s needs.
